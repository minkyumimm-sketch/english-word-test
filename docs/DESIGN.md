# 設計ドキュメント（Ver1）

## 1. 全体構成

```
Excel(.xlsm)        --[SheetJSで解析]-->        行×列の配列 -+
  excelLoader.js                                              |
                                                                +--> wordRowParser.js --> WordEntry[] --> wordRepository.js
Googleスプレッドシート --[CSV取得・パース]-->    行×列の配列 -+         （列判定・行検証）        |
  googleSheetLoader.js                                                                          v
                                                                                          testGenerator.js
                                                                                                  |
                                                                                                  v
                                                                                            TestSet[]
                                                                                                  |
                                                                                                  v
                                                                          resultView.js（プレビュー / 印刷用DOM構築）
```

- `main.js` が上記の流れをイベント駆動で配線するエントリーポイント。
- 画面表示専用ロジック（DOM操作）と、単語データの検索・出題生成ロジックを分離している。
  ロジック側（`wordRepository.js` / `testGenerator.js`）はDOMに依存しないため、
  将来的にテストを書く場合もブラウザなしで検証できる。
- データ入力元（Excel／Googleスプレッドシート）は「行×列の文字列配列」に変換するところ
  までがそれぞれの読込モジュールの責務で、そこから先（列判定・行の検証・WordEntry化）は
  `wordRowParser.js` に一本化している（Ver2.0）。詳細は3章。

## 2. データモデル

### WordEntry（1単語）

| フィールド | 型 | 内容 |
|---|---|---|
| rank | number | 順位（1始まり） |
| level | string | レベル名（例: "最重要レベル"）。空文字の場合あり |
| page | string | 掲載ページ（例: "p26-p27"）。表示には未使用（将来の拡張用に保持） |
| en | string | 英単語 |
| ja | string | 日本語訳 |

### TestSet（1回のテスト、または一括生成時の1日分）

| フィールド | 型 | 内容 |
|---|---|---|
| id | string | `"single"` または `"day1"`, `"day2"`... |
| label | string | 画面・印刷に表示する見出し |
| direction | `'en-ja'` \| `'ja-en'` | 出題方向 |
| rangeLabel | string | 順位範囲の表記（例: "1〜100"） |
| availableCount | number | 条件に合致した単語の総数（問題数より少ない場合は絞り込み結果の全件が出題される） |
| items | `{no, prompt, answer, rank}[]` | 出題内容。prompt/answerはdirectionにより英↔日が入れ替わる |
| poolItems | `{no, prompt, answer, rank}[]` | 出題範囲に存在する全単語（抽選前）。印刷レイアウト（列数・文字サイズ）の幅安全判定にのみ使う。ランダム出題で毎回違う単語が選ばれてもレイアウトを安定させるため（Ver2.7、6.4章参照） |
| dayNumber | number（`generateBatch`のみ） | 何日目か（1始まり）。生成予定パネルの注記文で使用（Ver2.1） |
| startRank | number（`generateBatch`のみ） | その日の開始順位（Ver2.1） |
| nominalEndRank | number（`generateBatch`のみ） | `chunkSize`から計算した本来の終了順位（Ver2.1） |
| actualEndRank | number（`generateBatch`のみ） | 実際に存在する単語の最大順位。`nominalEndRank`と異なる場合、データが尽きたことを示す（Ver2.1） |
| isShort | boolean（`generateBatch`のみ） | `actualEndRank < nominalEndRank`のとき`true`（Ver2.1） |

## 3. データ読込（Excel／Googleスプレッドシート）

Ver1では`excelLoader.js`が単独で「Excelファイル→単語データ」の全工程（シート選択、
列判定、行の検証）を担っていたが、Ver2.0でGoogleスプレッドシート読込
（`googleSheetLoader.js`）を追加するにあたり、**列判定・行の検証ロジックを
`wordRowParser.js`に切り出して共通化**した（既存の`excelLoader.js`へ無理に
実装を追加せず、責務を分離する方針のため）。

### 3.1 共通ロジック（wordRowParser.js）

- 入力は「行×列の文字列の2次元配列」（`rows[0]`=ヘッダー行、`rows[1..]`=データ行）。
  Excel・Googleスプレッドシートのどちらも、この形へ変換したうえで渡す。
- 1行目をヘッダーとして読み、セル文字列に「レベル」「順位」「ページ」「英語」「訳」
  （または「日本語」「意味」）を含む列を自動判定する（`detectColumns`）。見つからない列は
  既定位置（A=レベル, B=順位, C=ページ, D=英語, E=訳）にフォールバックする。
  → 既存Excelの列構成（`Data/`配下のサンプルで確認済み）にそのまま対応しつつ、
    列順が変わっても壊れにくくするための設計（Ver1から継続）。
- ヘッダー行の列数が2未満の場合は「列が不足しています」エラーを送出する（Ver2.0で追加。
  Googleスプレッドシート側で列構成が大きく崩れているケースを早期に検知するため）。
- 順位が数値でない行、英語または訳が空の行はスキップし、`skippedRows` として件数を返す。
- 有効な単語が1件も無い場合はエラーを送出する。
- 読み込んだ単語は rank 昇順にソートして保持する。
- `excelLoader.js`・`googleSheetLoader.js`のどちらも`parseRows(rows)`を呼び出すだけで、
  列判定・検証ロジックを重複実装しない。

### 3.2 Excel読込（excelLoader.js）

- シートは `Sheet1` / `単語` / `単語データ` / `データ` のいずれかの名前を優先して探し、
  なければ先頭シートを使う。
- SheetJSの`sheet_to_json({header:1})`で行×列の配列を取得し、`wordRowParser.parseRows()`
  に渡すのみ（Ver1からのロジック変更なし。責務の切り出しのみ）。

### 3.3 Googleスプレッドシート読込（googleSheetLoader.js, Ver2.0）

「ファイル→共有→ウェブに公開」したCSV（`docs.google.com`の公開CSVエンドポイント）を
`fetch()`で取得し、`utils/csv.js`（依存なしの簡易CSVパーサー）でパースしたうえで
`wordRowParser.parseRows()`に渡す。

#### URL変換（`toCsvUrl`）

利用者は「編集画面の通常URL」「ウェブに公開のURL」「既にCSVのURL」のいずれを貼っても
よい仕様のため、正規表現でURLの種類を判定し変換する（`googleSheetLoader.toCsvUrl`）。

| 入力URLのパターン | 変換後 |
|---|---|
| `/spreadsheets/d/{id}/edit...`（通常の編集URL） | `/spreadsheets/d/{id}/export?format=csv[&gid={gid}]` |
| `/spreadsheets/d/e/{publishedId}/...`（ウェブに公開のURL） | `/spreadsheets/d/e/{publishedId}/pub?output=csv[&gid={gid}]` |
| 既に`output=csv`/`format=csv`等を含むURL | そのまま使用 |
| `docs.google.com`以外のURL | そのまま使用（汎用の公開CSV URLとして扱う） |
| `gid`（シートのタブID） | `?gid=`・`#gid=`のいずれからも抽出し、変換後のURLへ引き継ぐ |

`docs.google.com/spreadsheets/d/e/...`という「ウェブに公開」専用のID（`2PACX-...`等）は、
通常の編集URLからは導出できない（Google側が公開操作時に別途発行するIDのため）ため、
既に`/d/e/`を含むURLはそのパターンとして扱い、`output=csv`を強制するに留めている。

#### CORSについて

Googleスプレッドシートの「ウェブに公開」CSVエンドポイントは、クロスオリジンの`fetch()`を
許可するレスポンスヘッダーを返す（一般的なWebアプリからの直接読み込みを前提に設計された
エンドポイントであるため）。本アプリはこの前提のもとfetchで直接取得する設計とし、
サーバー側プロキシは持たない（GitHub Pages上の静的サイトのみで完結させるため）。
「リンクを知っている全員が閲覧可」の共有設定のみでは、エンドポイントによっては
CORSヘッダーが付与されない・ログイン画面にリダイレクトされる場合があるため、
README.mdでは「ウェブに公開」の手順を明示的に案内している。

#### エラー分類

| 状況 | 検知方法 | ユーザー向けメッセージの主旨 |
|---|---|---|
| URL未入力・形式が不正 | fetch前のバリデーション | URLを入力/確認するよう案内 |
| 未公開のシートにアクセス（Googleが200 OKでログイン/エラーページのHTMLを返す） | レスポンス本文が`<html`等で始まるかをチェック（`isHtmlResponse`） | 「ウェブに公開」されているか確認するよう案内 |
| アクセス拒否（HTTP 401/403） | `res.ok`判定＋ステータスコード分岐 | 公開設定を確認するよう案内 |
| シートが見つからない（HTTP 404） | 同上 | URLを確認するよう案内 |
| その他のHTTPエラー（5xx等） | 同上 | 「CSVの取得に失敗しました（HTTP xxx）」 |
| 通信エラー（ネットワーク遮断・CORSブロック・名前解決失敗等） | `fetch()`が`TypeError`を投げる（ブラウザの仕様上、原因の詳細は取得不可） | 「通信エラー」として、ネットワーク・URL・公開設定の3点を確認するよう案内 |
| 列不足・データ0件 | `wordRowParser.parseRows()`が送出するエラーをそのまま伝播 | 3.1参照。Excel読込と共通のメッセージ |

いずれもPlaywrightでモックCSVサーバー（200/403/404/500・HTML応答・列不足・データ0件・
到達不能ポート）を用いて再現し、期待どおりのメッセージが表示されることを確認済み。

## 4. 出題生成（testGenerator.js）

- `generateSingle(words, settings)`: 単発テスト用に1つのTestSetを生成。`startRank`〜
  `endRank`の範囲から`questionCount`問を選ぶ（Ver1から変更なし）。
- `generateBatch(words, settings)`: **Ver2.1で終了順位ベースから作成日数ベースに変更**。
  `startRank`から`chunkSize`（1日あたりの順位幅）ごとに`dayCount`（作成日数）分だけ
  範囲を区切り、各範囲から`questionsPerChunk`問を選んで1日分＝1TestSetとする
  （既存VBA「一括100個ずつテスト」と同じ考え方を日数指定に拡張）。
  終了順位は使わない。該当単語が0件の範囲（データが完全に尽きた日）はスキップする
  （日番号はスキップしても詰めない。ループ変数`day`をそのまま`dayNumber`として使うため、
  実装上も自然にこの挙動になる）。
- 単語データが範囲の途中で尽きる場合（多くは最終日）、`rangeLabel`・`label`には
  「実際に存在する最大順位」（`actualEndRank`）を使う。ノミナルな終了順位
  （`nominalEndRank` = `start + chunkSize - 1`）より`actualEndRank`が小さい場合は
  `isShort: true`を各TestSetに付与し、UI側（生成予定・実際のラベル）で
  「〇日目は順位X〜Yが対象です」という注記を出す判断材料にする。
- 出題順が「ランダム」の場合は Fisher-Yates シャッフルを使用（`shuffle()`）。
- 生成された `items` の順序が、問題ページ・解答ページの両方でそのまま使われる
  （解答生成時に再シャッフルしない。プレビューと印刷も同じデータソース）。

### 4.1 生成予定プレビュー（Ver2.1）

複数日モードでは、「テストを生成する」ボタンを押す前に、日程ごとの範囲・問題数と
合計（対象語数・総出題数）を表示する「生成予定」パネルを設ける。

- `main.js`の`updateGenerationPreview()`が、フォームの現在値から`testGenerator.generateBatch()`
  を**実際の生成と全く同じ関数で**呼び出し、結果（`label`・`availableCount`・`items.length`・
  `isShort`等）をそのまま`formView.renderGenerationPreview()`に渡して描画する。
  プレビュー専用の見積もりロジックを別途持たない設計とすることで、
  「プレビューの件数」と「実際に生成した件数」がズレるバグを構造的に防いでいる
  （ランダム抽選で選ばれる単語そのものは呼び出しごとに変わるが、件数・範囲は
  条件が同じであれば常に一致する）。
- 再計算のトリガーは、開始順位・1日あたりの順位幅・作成日数・1日あたりの問題数・
  レベル指定の入力/変更イベント、および生成モードの切り替えイベント
  （`formView`の`onPreviewInputsChanged`ハンドラ経由）。単語データ未読込、または
  単発モードのときはプレビューを非表示にする。
- バリデーションは`validators.validateSettings()`をそのまま利用する。プレビュー時に
  単発モード専用の「終了順位」チェックが誤って働かないよう、
  `endRank < startRank`のチェックは`batchEnabled`が真のときはスキップする
  （画面上も終了順位欄自体を複数日モードでは非表示にしている）。

## 5. 画面・印刷（renderer/resultView.js）

- `.test-page` 要素をプレビュー用（`#previewArea`）と印刷用（`#printRoot`）の両方に
  同じ関数 `buildPageElement(testSet, kind)` で構築する。表示の見た目がズレないようにするため。
  内部的には、DOM構築そのものを担う`buildPageDom(testSet, kind, layout)`と、それに
  印刷1ページ収まり調整（`printFitting.fit`、6.4章）を組み合わせる`buildPageElement`に
  分かれている（Ver2.4）。
- 印刷用DOM（`#printRoot`）は常に「全TestSet × 問題/解答」を構築しておき、
  実際にどれを紙に出すかは `body[data-print-mode]` 属性とCSS（`print.css`）側で
  表示/非表示を切り替える。JSでDOMを都度作り直さない設計。
- 印刷モードは3種類:
  - `problem`: 問題ページのみ
  - `answer`: 解答ページのみ
  - `both`: 各日程ごとに「問題→解答」の順で連続印刷
- 各 `.test-page` に `page-break-after: always` を設定し、複数日分は1日ごとに改ページする。

## 6. 印刷レイアウトの自動最適化（layoutRules.js, Ver1.1）

問題数・単語/訳の文字数に応じて、文字サイズ・行間・列数・余白を自動決定する。
固定値の分岐ではなく、以下の2段階のルールで算出する（`layoutRules.computeLayout(items)`）。

### 6.1 問題数による基本レイアウト（COUNT_TIERS）

問題数の段階ごとに基本パラメータを表形式（`COUNT_TIERS`）で管理する。閾値や数値を
調整したい場合はこの表を編集するだけでよい。

| 問題数 | 列数 | 文字サイズ | 行間 | 余白 |
|---|---|---|---|---|
| 〜20問 | 1列 | 15pt | 2.3 | 16mm |
| 21〜40問 | 1列 | 13pt | 2.0 | 13mm |
| 41〜50問 | 1列 | 12pt | 1.8 | 11mm |
| 51〜60問 | 2列 | 12pt | 1.8 | 11mm |
| 61〜90問 | 2列 | 10.5pt | 1.6 | 9mm |
| 91問〜 | 2列 | 9.5pt | 1.45 | 8mm |

「41〜60問は必要に応じて2列」という要件は、50問を境に列数だけが変わる2行に
テーブルを分けることで表現している（41〜50は1列、51〜60は2列）。

### 6.2 文字数による補正（長い単語・長文訳への対応）

英単語・日本語訳それぞれの表示幅を概算し（全角文字=2、半角文字=1としてカウント。
`estimateDisplayWidth`）、A4の印刷可能幅・実際のフォントサイズ・列数・余白から
「1列に収まる概算の文字幅」を逆算する（`estimateColumnCapacityUnits`）。

固定の閾値（例:「24文字を超えたら長文」）ではなく、そのつどの文字サイズ・列幅から
逆算した収容可能幅と比較するため、6.1のテーブルの数値を変えても自動的に整合する。

1. 出題セット内で最も長い項目が、2列時の収容可能幅を超える場合は1列に切り替える。
2. 1列に切り替えてもなお収まらないほど極端に長い項目がある場合は、文字サイズを
   1段階（1pt）下げる（下限9pt。読みやすさを優先し、これより小さくはしない）。

問題ページ・解答ページは同じ `items`（同じ出題セット）から算出した同一のレイアウト値を
使うため、両ページで文字サイズ・列数がずれることはない。

### 6.3 実装上の注意点

- 用紙の物理的な余白（`@page { margin: 10mm 10mm; }`、`print.css`）は固定値のまま変更しない。
  「問題数に応じた余白の広さ」は `.test-page` 要素の `padding`（CSS変数 `--tp-padding`）で
  表現する。`@page` の余白を要素ごとに動的化するCSS機能（named page等）はブラウザ間の
  対応状況にばらつきがあり、印刷崩れのリスクを避けるため採用していない。
- 算出したレイアウト値は `resultView.js` の `buildPageElement()` が `.test-page` の
  インラインstyleにCSSカスタムプロパティ（`--tp-columns` / `--tp-font-size` /
  `--tp-line-height` / `--tp-padding` / `--tp-column-gap`）として設定し、
  `style.css`（画面プレビュー）・`print.css`（印刷）の両方が同じ変数を参照する。
  画面プレビューと印刷で見た目がずれない設計。
- 実データ（`Data/`配下の2000語）で計測したところ、英単語は最大15文字幅・日本語訳は
  95パーセンタイルで約22文字幅・まれに49文字幅程度（ふりがな相当の読みが本文に直接
  含まれているごく一部の項目）まで存在することを確認済み。6.2の補正ロジックは、
  この実データを用いたPlaywright自動テストで、複数の問題数パターン・長文パターンとも
  行のはみ出し（横方向オーバーフロー）が発生しないことを確認している。

### 6.4 印刷1ページ収まり調整（printFitting.js, Ver2.4／Ver2.5・Ver2.6で拡張）

6.1〜6.3の「問題数段階ベースのレイアウト」は、印刷結果が1ページに収まるかどうかまでは
考慮していなかった（例えば20問でも、1問あたりの行の高さが大きいため実際には2ページに
またがることがあった）。Ver2.4では、これに追加して「問題ページ・解答ページをそれぞれ
個別に実測し、はみ出す場合のみ・必要最小限だけ自動縮小する」調整を導入した。

要件の優先順位は次のとおり（ユーザー指定）。

1. 読みやすさ（最優先）
2. 可能なら1ページに収める
3. レイアウトの美しさ

「無理に1ページへ収めるために文字が小さくなり過ぎる実装」を避けるため、可読性を
守れる下限まで縮小しても収まらない場合は、縮小を打ち切って複数ページを許容する。

#### 実測の仕組み（測定サンドボックス）

`layoutRules.js`はDOM操作を行わない純粋関数という設計原則（ファイル冒頭のコメント参照）
を持つため、DOM計測を伴うこの調整は**別ファイル`printFitting.js`**に分離した。

- `index.html`に画面外の測定用要素`#printFitSandbox`を常設している
  （`position:absolute; visibility:hidden; left:-99999px; width:190mm;`）。
  `display:none`にはしていない点が重要（`display:none`だとレイアウト計算自体が
  行われず高さが常に0になるため）。
- `#printRoot`は画面表示では`display:none`（`style.css`）のため、印刷用DOMをそのまま
  計測に使うと高さが常に0になってしまう。そのため、候補レイアウトごとに実際の
  `.test-page`要素を`buildPageDom()`（同じ関数）で組み立て直し、
  `#printFitSandbox`へ一時的に追加して`getBoundingClientRect().height`で実測、
  測定後は取り除く、という手順を1候補ごとに繰り返す（`printFitting.measureHeightMm`）。
- サンドボックスの幅は190mm固定（`layoutRules.PRINTABLE_WIDTH_MM`と同じ）。画面プレビューの
  表示幅（`.preview-area`のCSSに依存し、ウィンドウ幅で変わり得る）とは独立させることで、
  常に実際の印刷時と同じ折り返しで高さを測る。
- 問題ページ・解答ページは内容（空欄プレースホルダー vs 実際の解答文字列）が異なり
  実測高さも変わり得るため、`buildPageElement(testSet, kind)`の呼び出しごとに
  個別に計測・調整する（要件の「それぞれ個別に計測」を、関数の呼び出し粒度がそのまま
  満たす自然な設計）。

#### 調整レバーと優先順位（Ver2.5で①②、Ver2.6で②を補強する仕組みを追加）

Ver2.4時点は縮小5種類（フォントサイズ・行間・余白・問題間隔・セクション間隔）のみ
だったが、実運用で「50問程度でも1ページに収まらない」「複数日分の一部の日だけ
1列になり、統一感がない」という問題が見つかったため、Ver2.5で**縮小に頼る前の
「レイアウト改善」を2段階追加**した。さらにVer2.6で、②の効果を高めるための
「長文の補足部分の縮小」（①と同じく常時適用のレイアウト改善）を追加した。
優先順位は次のとおり（数字が小さいほど先に試す）。

| 順序 | 種別 | 内容 | 対応するCSS変数/実装 | 可読性への影響 |
|---|---|---|---|---|
| ① | レイアウト改善（常時適用） | 問題ページのヘッダーを印刷専用で横並び化 | `print.css`の`.test-page[data-kind="problem"] .page-header`（後述） | なし（情報量は変えず配置のみ変更） |
| ①' | レイアウト改善（常時適用、Ver2.6） | 丸括弧書きの補足部分を印刷時のみ縮小 | `print.css`の`.item-supplement`（後述） | なし（文字は省略せず、補足部分のみ80%サイズ） |
| ② | レイアウト改善（条件付き） | 1列になっているページを2列に戻せないか試す（①'の縮小を織り込んだ幅で判定） | `printFitting.tryTwoColumns()` | なし（2列で収まる幅がある場合のみ採用） |
| ③ | 縮小 | 余白（padding） | `--tp-padding`（16〜6mm、1mm刻み、下限6mm） | 小 |
| ④ | 縮小 | セクション間隔 | `--tp-section-gap-factor`（1.0〜0.4、0.1刻み） | 小 |
| ⑤ | 縮小 | 問題間隔 | `--tp-row-gap-factor`（1.0〜0.3、0.1刻み） | 小 |
| ⑥ | 縮小 | 行間 | `--tp-line-height`（0.05刻み、下限1.4） | 中 |
| ⑦ | 縮小 | フォントサイズ | `--tp-font-size`（0.5pt刻み、下限9pt） | 大（最後の手段） |

①・①'は「常に適用される」レイアウト改善（`fit()`のどのケースでも無条件に有効）、
②は「1列を2列に戻せないか」を条件付きで試すレイアウト改善で、いずれも文字を
小さくせずに済む余地を先に使い切るという位置づけのため、③〜⑦（Ver2.4からの
5つの縮小レバー）よりも常に先に試す。1つのレバー（または段階）を使い切ったら
次に進む「順次使い切り方式」は従来どおり。全レバーを下限まで使っても収まらない場合は、
それ以上縮小せず複数ページを許容する（`fit()`の戻り値`fitsOnePage: false`）。

##### ①ヘッダーの印刷専用横並び化

問題ページのヘッダーは、画面プレビューでは従来どおり
「タイトル＋出題方向」→「順位範囲／問題数」→「名前欄・点数欄（罫線付き）」の3段
だが、印刷時のみ「タイトル＋名前欄＋点数欄」を1行に、「出題方向＋順位範囲／問題数」を
1行にまとめ、2段に圧縮する（罫線付きの名前欄ブロックが丸ごと1段減る）。

```
【画面プレビュー（変更なし）】      【印刷（Ver2.5）】
英単語テスト        英語→日本語     英単語テスト   名前：＿＿＿＿＿＿  点数：　／50
順位1〜50／全50問                   英語→日本語   順位1〜50／全50問
─────────────────────
名前：＿＿＿＿＿＿  点数：　／50
─────────────────────
```

DOM構造は画面・印刷で共通のまま変更していない（`resultView.js`の`buildPageDom`は
従来どおり`.page-header-top`→`.page-meta`→`.page-name-line`の順で子要素を作る）。
`print.css`側だけで、`.page-header-top`と`.page-name-line`を`display:contents`に
することで、中の子要素（タイトル・出題方向・名前・点数）を`.page-header`直下の
flexアイテムとして扱えるようにし、`order`と`flex-basis:100%`（改行位置の強制）を
使って印刷時だけ配置を組み替えている。DOM構造そのものは変えないため、画面プレビュー
（`style.css`側は無変更）には一切影響しない。

**重要な注意点（実装時に踏んだ落とし穴）**: `printFitting.js`の高さ実測は
`#printFitSandbox`（画面内、`visibility:hidden`で隠しているだけ）で行うため、
「印刷時（`@media print`）だけ」に上記のヘッダー圧縮ルールを書くと、実測時には
まだ画面用の3段ヘッダーのままで計測されてしまい、「実際に印刷すると2段で収まるのに、
実測では3段分の高さとして計算されて余計に縮小してしまう」という不整合が起きる
（Playwrightの検証で実際に検出した）。対策として、`style.css`側に
`#printFitSandbox .test-page[data-kind="problem"] .page-header`という、
`print.css`の該当ルールと**まったく同じ内容**のルールを複製し、サンドボックス内では
メディアクエリに関係なく圧縮後のヘッダーで計測されるようにしている。CSSのメディア
クエリは要素単位で「印刷時だけ」を再現する手段がないため、この複製は意図的なもの。
`print.css`側のヘッダー関連ルールを変更する場合は、`style.css`の複製側も必ず
揃えること（両ファイルにその旨のコメントを記載済み）。

解答ページには元々罫線付きの名前欄が無い（`.page-name-line`は問題ページのみ生成）ため、
この変更の対象外（従来どおりタイトル＋出題方向の1行、順位範囲／問題数の1行の2段のまま）。

##### ②2列レイアウトへの復帰（`tryTwoColumns`）

`layoutRules.computeLayout()`が算出した基本レイアウトが1列の場合
（問題数が少ないため元々1列のティア、または6.2の長文安全策で2列から1列に
落とされた場合の両方を含む）で、かつ実測してみると1ページに収まらない場合、
数値を縮小する前に「2列に戻せないか」を試す（`printFitting.tryTwoColumns`）。

- 6.2と同じ`estimateColumnCapacityUnits`の計算式を使い、まず基本レイアウトの
  フォントサイズのまま2列が収まるか確認する。収まらない場合のみ、フォントサイズを
  0.5pt刻みで下限（9pt）まで下げながら、2列で収まる組み合わせを探す。
- 2列候補が見つかった場合、実測して1列のままより高さが低くなることを確認したうえで
  採用する（低くならない場合は1列のまま、後続の縮小レバーに進む）。
- 2列への復帰で採用した文字サイズは、そのまま後続の縮小レバー（③〜⑦）の
  出発点として引き継がれる（2列にしてもまだ収まらない場合は、そこから通常どおり
  余白・間隔・行間・フォントサイズの縮小を続ける）。
- 2列にしても幅が収まらないほど極端に長い項目がある場合（`tryTwoColumns`が
  `null`を返す場合）は1列のまま③以降に進む。これは「読みやすさ最優先」の設計どおりの
  動作である（11章の既知の制約を参照。Ver2.6で③の補足部分縮小を追加したことで、
  丸括弧書きの長い訳文の多くはこの分岐に入らず2列を維持できるようになった）。
- 問題数段階のティア自体が決める列数（6.1）や、6.2の長文安全策そのものは変更していない。
  `printFitting.js`はあくまで「1列という出発点に対して、2列に戻せる余地がないか」を
  追加で試すだけであり、既存の判定ロジックの責務を奪うものではない。

##### ②の実効性を高める: 長文の補足部分（丸括弧書き）を印刷時のみ縮小（`splitSupplementSegments`, Ver2.6）

これは③〜⑦のような独立した縮小レバーではなく、**①と同様「常時適用される、印刷専用の
レイアウト改善」**であり、その効果が②（2列復帰）の幅判定にも反映される、という位置づけ。
②の2列復帰を試しても、訳文が長すぎて2列に戻せないケースが実データで見つかった
（例:「（Shall I〜?/Shall we～？で）〜しましょうか。」のような、丸括弧書きの
補足説明を含む訳文）。この種の補足説明は、印刷された解答用紙では本体の訳
（この例では「〜しましょうか。」）ほど重要度が高くない場合が多いため、**情報を
省略せずに、補足部分だけ印刷時に少し小さいフォントサイズで表示する**ことで
横幅を抑える改善を追加した（要件として「文字を省略するのではなく縮小する」方式が
明示的に選ばれている。解答用紙で文字が欠けると採点の正確性を損なうため）。

- `layoutRules.splitSupplementSegments(text)`が、丸括弧（全角`（）`・半角`()`の
  どちらにも対応）で囲まれた部分を「補足」、それ以外を「本文」として文字列を
  分割する（DOM操作をしない純粋関数）。ネストした括弧は非対応（実データの範囲では
  発生しないため、シンプルさを優先）。
- `resultView.js`の`buildSupplementNodes()`が、分割結果をもとに補足部分だけ
  `<span class="item-supplement">`でラップしたノード列を作る。`item-prompt`・
  `item-answer`の両方に適用している（英単語側に丸括弧の文法注記が付くケースにも
  対応するため）。**文字列そのものは一切変更しない**（`textContent`は元の文字列と
  完全に一致する）ため、省略による情報欠落は起きない。
- `.item-supplement`のフォントサイズ縮小は`print.css`にのみ書き、`style.css`
  （画面プレビュー）側には書かない。そのため画面では常に通常サイズのまま表示される。
- 縮小比率は`layoutRules.js`の`SUPPLEMENT_WIDTH_RATIO = 0.8`（ブラウザ標準の
  `<small>`要素の縮小率、約83%に近い値）。`print.css`/`style.css`の
  `.item-supplement { font-size: 0.8em; }`と数値を必ず一致させること。
- 2列復帰（②）の幅判定にこの縮小を織り込むため、`layoutRules.js`に
  `estimateDisplayWidthWithSupplement()`（補足部分の概算幅をこの比率で割り引く）・
  `computeMaxItemWidthPrintAware()`を追加し、`printFitting.tryTwoColumns()`は
  従来の`computeMaxItemWidth`ではなくこちらを使うよう変更した。`layoutRules.
  computeLayout()`自体（screen/print共通の基本判定）は変更していない
  （②で触れた「既存の判定ロジックの責務を奪わない」方針を踏襲）。
- ヘッダー横並び化（①）と同様の理由で、`#printFitSandbox`での高さ実測にも
  この縮小を反映させる必要があるため、`style.css`に
  `#printFitSandbox .item-supplement { font-size: 0.8em; }`という、
  `print.css`と全く同じ内容のルールを複製している。

**検証結果**: 実データで丸括弧の長文訳が1件だけ含まれる範囲（順位500〜554、
55問中1問）を検証したところ、Ver2.5時点では2列に戻せず1列・9pt（下限）でも
296.0mmで収まらなかったページが、Ver2.6では2列・9pt・273.5mmに収まるように
なった（4日分バッチの実PDFページ数も、対象日を含めて4ページ＝全日程1ページずつに
改善したことを確認済み）。

一方、別の実データ範囲（順位1500〜1554）で見つかった「訳文にカタカナの読みが
括弧なしで直接連結されている」ケース（例:「（発表などで投影する）スライド
ハッピョウトウエイ」。「発表などで投影する」は括弧内の補足として縮小されるが、
末尾の「ハッピョウトウエイ」は括弧で囲まれていないため縮小対象にならない）は、
今回の改善の対象外である。これは11章に記載済みの別の既知の制約（読みが直接
連結されたデータ）に起因するもので、区切り文字が無いため機械的に「本文」と
「補足」を判別できない。このケースは引き続き1列・可読性下限まで縮小しても
収まらず2ページになることがある。

##### 各ページが独立最適化されることについて（複数日分バッチ生成時）

`fit()`は`buildPageElement(testSet, kind)`の呼び出しごと（＝日ごと・問題/解答ごと）に
実行されるため、①②③〜⑦のすべてが常にページ単位で独立して計算・適用される。
複数日分のバッチ生成で、日によって列数や縮小率が異なって見える場合があるが、
それは各日の単語の長さ（②の幅安全チェック）や実際の折り返しの違いにより、
各ページが個別に最適な結果へ到達した結果であり、日をまたいで共有された状態や
キャッシュはない（`printFitting.fit`はモジュールスコープに状態を持たない純粋な
呼び出しごとの計算）。

##### ランダム出題時の列数・文字サイズの安定化（`poolItems`, Ver2.7）

「同じ設定で複数日分を繰り返し生成すると、生成するたびに違う日が1列になる」という
不具合報告を受けて調査した結果（詳細は後述の調査手順を参照）、**fit()・計測サンドボックス・
非同期処理には一切の問題がなく**、原因は「出題順＝ランダム」時に`testGenerator.js`の
`shuffle()`（`Math.random()`使用）が生成のたびに異なる単語を抽選しており、②の幅安全
チェック（`layoutRules.computeLayout`/`printFitting.tryTwoColumns`）が**実際に抽選された
単語だけ**を見て列数を決めていたため、「たまたま長い単語が抽選に含まれた回だけ1列になる」
という、データ内容に起因する（バグではない）挙動だったことが判明した。

要望は「単語の抽選自体はランダム性を維持したいが、列数・フォントサイズという印刷レイアウトは
毎回同じ結果にしたい」というものだったため、**列数・文字サイズの安全判定の基準を「実際に
抽選された単語」から「その出題範囲に存在する全単語」に変更**した。

- `testGenerator.js`の`generateSingle`/`generateBatch`が、各TestSetに`items`（実際に
  出題される項目）に加えて`poolItems`（その順位範囲でクエリ条件に合致する全単語。
  抽選前の`filtered`をそのまま`buildItems()`に通したもの）を新たに持たせる。
- `layoutRules.computeLayout(items, widthSourceItems)`に第2引数を追加。省略時は
  `items`を使う（後方互換）。行数に基づく段階（COUNT_TIERS）の判定は引き続き`items.length`
  （実際に出題される問題数）を使うが、幅の安全判定（`computeMaxItemWidth`）だけは
  `widthSourceItems`（＝`poolItems`）を使う。
- `printFitting.tryTwoColumns(widthSourceItems, baseLayout)`も同様に、幅判定の入力を
  呼び出し元（`fit()`）から渡された`testSet.poolItems || testSet.items`に変更。
- `resultView.js`の`buildPageElement`は`layoutRules.computeLayout(testSet.items,
  testSet.poolItems)`と呼び出すよう変更。

この結果、実際に抽選された単語がたまたま短くても、列数・フォントサイズは「その範囲に
存在し得る最長の単語」を基準にした安全側の値になる。**抽選結果によっては余白が
やや多めに残ることを許容するトレードオフ**（ユーザー確認済み）。単発モード
（`generateSingle`）でも同じ仕組みを適用しており、出題範囲に対して出題数が少ない
（＝抽選の余地がある）場合は同様に安定化される。

###### 調査手順（再発時の切り分けに使えるよう記録）

1. **`printFitting.fit`と`layoutRules.computeLayout`をその場でラップして全呼び出しを
   ログ化**し、「出題順＝順位順（決定的）」で同一設定を10回連続生成 →
   毎回100%同じ結果（fit()呼び出し回数・列数とも完全一致）。fit()自体・サンドボックス・
   非同期処理を疑う根拠は無いことをまず確認した。
2. 「出題順＝ランダム」に切り替えて同条件で10回生成 → 1列になる日が実行のたびに変化
   （報告内容と同じパターンを再現）。ただし「1日あたりの問題数」と「1日あたりの順位幅」を
   同じ値にする（＝抽選の余地を無くす）と、ランダムでも安定することを確認 →
   「抽選そのもの」に依存した現象であるとここで切り分けられた。
3. **決定的な証明**: ランダムモードで同じ設定で2回生成し、`state.getTestSets()`から
   ある日の実際の単語配列を比較 → 55語中54語の位置で中身が異なっていた（「同じ設定」
   であっても実際のデータは別物であることを直接証明）。続けて、片方の回で実際に選ばれた
   単語配列を**固定入力**として`computeLayout()`→`fit()`を`testGenerator`を経由せず
   直接10回連続実行 → 高さ(mm)まで完全に同一。入力が同じであれば出力も完全に同じ
   であることを証明した。
4. **相関の検証**（データ内容が原因と言えるかの最終確認）: 同一設定で10回生成し、
   20日×10回＝200件について「1列になったか」と「その日の最長項目の実測幅が2列の
   許容幅（下限フォントで計算）を超えているか」を突き合わせたところ、**200件中32件が
   1列になり、そのすべてで実測幅が許容幅を超過（矛盾0件）**。合計文字数では説明が
   付かないケース（1列の日より2列を維持した日の方が総文字数が多い、等）があることも
   確認したが、これは「合計文字数」ではなく「最長1項目の幅」で列数が決まる設計のため
   正常（6.2章参照）。

この4段階の切り分けにより、原因が処理タイミングではなくデータ内容（ランダム抽選）に
あることを、推測ではなく実測で確定させたうえで、`poolItems`による安定化を実装した。

###### テスト結果

ランダム出題・20日分・1日あたり100語の範囲から55語を抽選、という設定で10回連続生成し、
全40ページ（20日×問題/解答）の列数・フォントサイズ・行間・余白・実測高さ(mm)を比較した
ところ、**10回すべてで完全に一致**した。一方、各回で実際に選ばれた単語の並び（fingerprint）は
10回とも異なっており、**単語の抽選自体のランダム性は損なわれていない**ことも確認済み。

#### CSS変数の実装（既存の calc() に factor を掛ける形）

`--tp-row-gap-factor` / `--tp-section-gap-factor` は、既存の`--tp-line-height`
（問題間隔）・`--tp-padding`（セクション間隔）由来の`calc()`式に**掛け算で追加**した
（`style.css`の`.item-row`/`.page-header`、`print.css`の対応箇所）。

```css
/* 問題間隔（行間由来のpaddingに掛ける） */
padding: calc((var(--tp-line-height, 1.6) - 1) * 0.5em * var(--tp-row-gap-factor, 1)) 0;

/* セクション間隔（paddingに掛ける） */
margin-bottom: calc(var(--tp-padding, 10mm) * 0.5 * var(--tp-section-gap-factor, 1));
```

factorの既定値は両方とも1のため、`printFitting`が縮小しない（=1のまま）場合は
既存のVer1.1時点の見た目・計算式と完全に一致する。これにより「すでに1ページに
収まっている場合は変更しない」という要件を、特別な分岐を書かずに自然に満たしている。

#### ページ高さの目標値

A4縦・`@page { margin: 10mm 10mm; }`（`print.css`、固定）から、印刷可能高さは
`297mm - 10mm*2 = 277mm`。画面計測と実際の印刷ラスタライズの間の微小な誤差
（フォントヒンティング等）を吸収するため、安全マージン3mmを引いた**274mm**を
実際の目標値（`TARGET_HEIGHT_MM`）としている。

## 7. 前回設定の自動保存・復元（Ver1.2）

最後に使用したテスト条件一式（フォーム入力値・印刷対象）をlocalStorageに保存し、
次回起動時に自動でフォームへ復元する。既存の単語データ復元（`state.persistWordsToLocalStorage`
等）と同じ「`state.js`が永続化を担当し、他モジュールはgetter/setter経由でのみ触れる」
という設計に自然に統合している。

### 7.1 保存する値

`formView.readRawSettings()` が返す形状（バリデーション前の生の入力値）をそのまま
保存する。正規化・検証済みの値ではなく生の文字列を保存することで、フォームへの
復元（`input.value = ...`）が単純な代入だけで済む。

```
{ testTitle, startRank, endRank, levelFilter, direction, order,
  questionCount, batchEnabled, chunkSize, questionsPerChunk, printMode }
```

`printMode`（問題のみ/解答のみ/問題→解答）はフォーム外（`resultView.js`側）の値のため、
保存時に `main.js` の `saveCurrentSettings()` が両者をマージしてから1つのオブジェクトとして
書き込む。

### 7.2 保存タイミング

- **テスト生成成功時**（`handleGenerate`内、バリデーション通過かつ該当単語が1件以上ある場合）
- **印刷対象セレクトの変更時**（`resultView`の`onPrintModeChange`コールバック経由）

入力欄の変更ごと（キー入力毎）には保存しない。理由は2つ:
1. 保存回数が過剰になる。
2. バリデーション前の中途半端な値（例: 終了順位を消して入力し直している途中の空欄）を
   保存してしまうと、次回起動時にエラー状態から始まってしまう。

### 7.3 復元タイミングと順序

`main.js`の`DOMContentLoaded`ハンドラ内で、必ず
`restorePersistedWords()` → `restorePersistedSettings()` の順に呼ぶ。

レベル指定(`levelFilter`)は選択肢がExcelデータに依存するため、単語データの復元
（`<select>`への選択肢投入）が先に完了していないと、保存されていた値を正しく
選択できない。`formView.setFormValues()`側でも、現在の選択肢に存在する値だけを
復元する防御を入れている（Excelが別ファイルに差し替わり、以前のレベル名が
存在しなくなるケースを考慮）。

生成モード（単発/一括）を復元する際は、ラジオボタンの`checked`を直接書き換えるだけでは
`change`イベントが発火せず表示欄の出し分けが更新されないため、
`formView`内の`applyModeVisibility()`を復元処理から明示的に呼び出している。

### 7.4 データ消去との関係

「読み込んだデータを消去」ボタンは単語データ（`wordTestApp.lastWords.v1`）のみを消去し、
テスト条件の保存（`wordTestApp.lastSettings.v1`）はそのまま残す。単語データとテスト条件は
別の関心事であり、Excelを読み込み直す運用（先生がデータを更新した際の再読込等）のたびに
毎回条件を入力し直す必要がないようにするための設計判断。

## 8. GitHub Pagesでの公開（Ver1.3、Ver2.3で構成を変更）

詳しい公開手順はREADME.mdの「12. GitHub Pagesでの公開（管理者向け）」を参照。
ここでは設計判断の経緯のみ記す。

### 8.1 Ver1.3時点の方式と、Ver2.3での見直し

Ver1.3では、アプリ本体を`src/`フォルダにまとめたまま、GitHub Actions
（`actions/upload-pages-artifact`で`src/`をそのままアーティファクト化）**のみ**で
公開する方式を採用していた。検討時点では以下の理由で「リポジトリ直下への移動」を
不採用としていた。

- `server.js`のROOT設定や、CLAUDE.md/README.mdに記載済みのフォルダ構成、ローカル
  起動手順（`src/index.html`をダブルクリック）まで広範囲に変更が必要になり、影響範囲が
  大きい。
- GitHub Actionsで`src/`を直接公開すれば、公開用ファイルの複製を持たずに済む。

しかし実際の運用で、GitHub PagesのSource設定が「Deploy from a branch」
（GitHub Pagesの標準的な設定方法で、多くの場合こちらがデフォルト/最初に選ばれやすい）
になっていると、**Actionsワークフローが使われず**、リポジトリ直下がそのまま配信対象に
なることが判明した。この場合、直下に`index.html`が存在しないため、GitHub Pages標準の
Jekyll処理が`README.md`をホームページとして自動表示してしまい、
`https://<user>.github.io/<repo>/`を開いてもアプリが起動しないという問題が発生した。

この問題は「Source を GitHub Actions に設定する」ことでも回避できるが、**Source設定を
利用者側の運用に依存させず、どちらのSource設定でも確実にアプリが起動するようにする**
ことを優先し、Ver2.3でアプリ本体（`index.html`/`css`/`js`/`lib`）をリポジトリ直下へ
移動した。「既存機能を壊さない」ことを最優先しつつ、当初の想定より影響範囲を丁寧に
洗い出した結果、次の対応で移行コストを最小化できると判断した。

- `server.js`: `ROOT`を`path.join(__dirname, "src")`から`__dirname`に変更するのみ
  （配信対象を`index.html`/`css`/`js`/`lib`のトップレベル項目に限定するチェックを追加し、
  `Data/`等を誤って配信しないようにした）。
- `index.html`内の相対パス（`css/style.css`、`js/state.js`等）は、`index.html`と
  `css/`/`js`/`lib/`を**まとめて**リポジトリ直下へ移動したため、パス文字列自体は
  変更不要だった（相対的な位置関係が保たれるため）。
- `.github/workflows/deploy-pages.yml`: 直下の`index.html`/`css`/`js`/`lib`のみを
  一時ディレクトリへステージングしてから公開するよう変更（`Data/`・`docs/`等を公開対象
  から除外する目的は維持）。
- リポジトリ直下に`.nojekyll`（空ファイル）を追加し、「Deploy from a branch」使用時も
  Jekyll処理を無効化した（README.md等が余計に変換・公開されるのを防ぐ）。

### 8.2 相対パスについて

`index.html`および配下のCSS/JSは、もともと `css/style.css` `js/state.js` のような
**ルート相対（`/`始まり）ではない相対パス**で統一されていたため、GitHub Pagesの
プロジェクトサイト（`https://<user>.github.io/<repo>/` のようにドメイン直下ではなく
サブパス配下で配信される）でも変更なしでそのまま動作する（Ver1.3から変更なしの前提）。
今後ファイルを追加する場合も `/` 始まりの絶対パスは使わないこと。

### 8.3 公開範囲

GitHub Actions経由（`.github/workflows/deploy-pages.yml`）で公開する場合、ワークフローは
`index.html`/`css`/`js`/`lib`のみを一時ディレクトリへステージングしてからアーティファクト化
するため、`Data/`（単語データExcel）・`docs/`・`README.md`等は公開サイトに含まれない。
Excelは利用者が自分の端末から選択して読み込む方式のままであり、この設計は変更していない。

「Deploy from a branch」で公開する場合は、リポジトリのブランチの中身がそのまま配信される
ため、`Data/`が**コミットされていれば**Pages経由でも配信されてしまう。本プロジェクトでは
`Data/`を`.gitignore`に登録し、通常操作ではコミットされないようにしている。

いずれの方式でも、これはあくまで「Pages経由で配信されるかどうか」の話であり、リポジトリ
自体をPublicにした場合、`Data/`がコミットされていればGitHubのリポジトリ画面（コード閲覧・
git履歴）からは閲覧可能になる点は別問題として残る（README.md「12. GitHub Pagesでの公開」
内に注意書きを記載）。

## 9. データ読込方式の保存・復元（Ver2.0）

最後に使用したデータ読込方式（`excel`/`googleSheet`）と、入力したGoogleスプレッドシート
URLをlocalStorageに保存し、次回起動時にフォームへ復元する。7章（前回設定の自動保存・復元）
と同じ設計方針（`state.js`が永続化を担当し、他モジュールはgetter/setter経由でのみ触れる）
を踏襲しているが、以下の理由から7章の`wordTestApp.lastSettings.v1`とは**別のキー**
（`wordTestApp.lastLoadMethod.v1`）に分離して保存している。

- テスト条件（順位範囲・出題方向等）とデータ読込方式は関心事が異なる
  （前者は「どの単語をどう出題するか」、後者は「単語データをどこから取得するか」）。
- 読込方式の保存タイミングは「データの読込に成功した時」であり、テスト条件の保存タイミング
  （「テスト生成に成功した時」）とは異なるため、同一オブジェクトに混在させると
  保存タイミングの制御が複雑になる。

### 9.1 保存タイミング

- Excelファイルの読込に成功した時（`method: "excel"`として保存）
- Googleスプレッドシートの読込に成功した時（`method: "googleSheet"`として、入力された
  URLも合わせて保存）

Excel読込時にも、それまで入力されていたGoogleスプレッドシートのURL（フォームの入力欄の
現在値）は消さずに保持する（`main.js`の`saveLoadMethod()`）。同一セッション内・次回起動時に
Googleスプレッドシートへ切り替えたとき、URLを再入力する手間を減らすため。

### 9.2 復元時の注意点

単語データそのもの（`wordTestApp.lastWords.v1`）は、データ読込方式に関わらず既存の
`restorePersistedWords()`がそのまま復元する（読込元によらず、パース後は同じ`WordEntry[]`
形式になるため、Ver1からのロジック変更は不要）。読込方式の復元（`restorePersistedLoadMethod()`）
は、ラジオボタンとURL入力欄というUI状態を復元するのみで、**URLへの再アクセス（fetch）は
行わない**（Excel読込がアプリ起動のたびにファイルを自動で再読込しないのと同じ考え方。
ネットワークアクセスを伴う処理を、明示的なユーザー操作なしに自動実行しない）。

単語データの復元メッセージ（「前回読み込んだデータを復元しました」）の案内文は、
保存されていた読込方式に応じて「Excelを更新した場合は再度ファイルを選択してください」
「スプレッドシートを更新した場合は「読み込む」を押し直してください」を出し分けている。

## 10. 現在の教材表示（Ver2.2）

条件設定エリアの直前（データ読込エリア直下）に、常に表示される「現在の教材」パネルを
設ける。目的は、実際にテスト生成へ使われる教材（Excelファイル名／Googleスプレッドシート名）
と、それが今どちらの方式で読み込まれたものかを一目で分かるようにし、教材の取り違えを
防ぐこと。

### 10.1 表示のもとになるデータ

新しいlocalStorageキーは追加していない。教材名は、既存の`wordTestApp.lastWords.v1`が
持つ`sourceFileName`フィールド（Ver1から存在。従来は単に「読み込み状況メッセージ」の
文言生成にしか使っていなかった）をそのまま「現在の教材」パネルの表示名として再利用する。
`state.getSourceFileName()` / `state.updateSourceFileName()`が単一の情報源であり、
教材表示・復元メッセージ・localStorage保存のすべてがこの値を参照する。

### 10.2 教材名の決定ロジック

- **Excel**: 読み込んだファイル名（`File.name`）をそのまま使う（Ver1から変更なし）。
- **Googleスプレッドシート**: 3段階のフォールバックで決める（`googleSheetLoader.js`）。
  1. `deriveFallbackName(url)`: URLだけから同期的に分かる名称。Googleスプレッドシートの
     URL（`docs.google.com/spreadsheets/...`）はIDしか含まれず名称にならないため`null`を
     返す。Google以外の汎用CSV URL（例: `https://example.com/words.csv`）の場合は、
     URLの末尾（ファイル名相当）を返す。
  2. `fetchTitleBestEffort(url)`: 「ウェブに公開」されたURL（`/spreadsheets/d/e/{id}/...`
     パターン）の場合のみ、公開HTMLビュー（`.../pubhtml`）を追加で`fetch()`し、
     `<title>`タグからスプレッドシートのタイトルを抽出する（末尾の
     「 - Google スプレッドシート」等の定型文言は除去）。取得できない・失敗した場合は
     例外を投げず`null`を返す（データ読込の成否には一切影響させないため）。
  3. どちらも得られない場合は既定ラベル「Googleスプレッドシート」を使う。

### 10.3 表示更新のタイミング（2段階更新）

タイトル取得（10.2の2.）はCSV本体の取得とは別の追加のfetchであり、失敗もあり得るベスト
エフォート処理のため、**データ読込の完了を待たせない**設計にしている。

1. CSV取得・パースが成功した時点で、即座に「1.のフォールバック名（またはURL由来の名称）」
   で「現在の教材」パネル・読込成功メッセージ・localStorage保存を更新する
   （`main.js`の`handleLoadSheet()`）。
2. その後、非同期でタイトル取得を試み（`upgradeGoogleSheetTitle()`）、実際のタイトルが
   取得でき、かつ1.で表示した名称と異なる場合のみ、パネル表示・`state`・localStorageを
   タイトルへ差し替える。取得中にユーザーが別の教材を読み込み直していた場合
   （`state.getSourceFileName()`が1.の時点の値と一致しない場合）は上書きしない
   （古い非同期処理の結果で新しい状態を壊さないためのガード）。

読込成功メッセージ（`#loadStatus`）自体は、後からタイトルが取得できても再表示・書き換えは
しない（一度読んだ通知の文言が後から変わるとかえって分かりにくいため）。「現在の教材」
パネルは常に最新の状態を反映する常設インジケーターとして、そちらだけを更新する。

### 10.4 未読込時・データ消去時

「教材未読込」と表示する。アプリ起動直後（`main.js`の`DOMContentLoaded`冒頭で
`formView.renderMaterialInfo(null)`を呼ぶ）と、「読み込んだデータを消去」ボタン押下時
（`handleClearData()`）の2箇所で明示的にこの状態にする。

### 10.5 検証時に見つけた副次的な不具合の修正

Googleスプレッドシート読込→Excelへ切替→**直前と同じファイル**を再選択、という操作を
Playwrightで検証したところ、`<input type="file">`の`change`イベントが発火せず
「現在の教材」パネルが更新されない不具合が見つかった。これはブラウザの一般的な仕様
（同一ファイルの再選択では`input.value`が変化しないためchangeイベントが発火しない）に
起因するもので、Ver1から潜在していた（本来はExcelを編集後、同名で再保存して選び直す
運用でも発生し得た）。`formView.js`の`fileInput`の`change`リスナー内で、処理後に
`els.fileInput.value = ""`へリセットすることで、同じファイルを連続選択しても確実に
`change`が発火するよう修正した。

## 12. 読み込んだ単語データ一覧のCSV書き出し（Ver2.8）

「読み込んだデータを消去」ボタンの隣（`#loadActions`内）に「単語データをCSVで書き出す」
ボタンを設け、`state.getWords()`が保持する単語データ一覧（絞り込み前の全件。テスト条件の
順位範囲・レベル指定によるフィルタは適用しない）をCSVファイルとしてダウンロードできる
ようにした。

### 12.1 CSV文字列の組み立て（`utils/csv.js`の`stringify`）

既存の`csv.parse()`（CSV文字列→行×列の配列）の逆変換として`csv.stringify(rows)`を追加した。
カンマ・ダブルクォート・改行を含むフィールドのみダブルクォートで囲み、内部のダブルクォートは
`""`にエスケープする（RFC4180相当、`parse()`と対称的な実装）。行区切りはExcel互換のCRLFに
している。ヘッダー行は`["順位","レベル","ページ","英語","訳"]`とし、`wordRowParser.js`の
列判定キーワード（`レベル`/`順位`/`ページ`/`英語`/`訳`）とそろえてあるため、書き出したCSVを
そのままExcel/Googleスプレッドシート経由で読み込み直しても列判定が機能する。

### 12.2 ファイル名の決定（`utils/csv.js`の`buildFileName`）

ファイル名の初期値は、現在の「単語テスト名」から機械的に組み立てる。呼び出し側
（`main.js`の`handleExportCsv()`）は`state.getSourceFileName()`を直接呼ばず、必ず
`state.getCurrentTestName()`を経由する（詳細は13章「単語テスト名」の項を参照。
`getCurrentTestName()`自体は、拡張子を除いた「単語テスト名」を返す）。
新しいlocalStorageキーや状態は追加していない（ダウンロードボタン押下時に毎回その時点の
値を読むだけなので、教材を読み込み直せば次のダウンロード時には自動的に新しい名前が
使われる。教材変更を検知して何かを更新するイベント配線は不要）。

変換手順は次の通り（`buildFileName(materialName)`）。

1. 末尾の既知拡張子（`.xlsx`/`.xlsm`/`.xls`/`.csv`、大文字小文字を無視）を除去する
   （`utils/csv.js`の`stripKnownExtension()`を呼ぶだけで、拡張子リストの実体はここ
   一箇所にしか持たない。`state.getCurrentTestName()`が渡す時点で既に除去済みのため
   実質的には何もしないが、`buildFileName()`単体で他の呼び出し元から生の教材名を
   渡されても安全なように、防御的に呼んでいる）。
2. Windowsで使用できない文字（`\ / : * ? " < > |`）を除去する。
3. 前後の空白を除去する。結果が空文字列になった場合は既定名「単語データ」にフォールバックする。
4. 最後に`.csv`を付与する。

ダウンロード自体は`main.js`の`handleExportCsv()`が、`Blob`＋一時的な`<a download>`要素の
クリックという標準的な手法で行う。ファイル名はブラウザの保存ダイアログ（対応環境の場合）で
ユーザーが自由に変更できる、通常の`download`属性の挙動をそのまま使っているだけであり、
アプリ側でダイアログ自体を制御してはいない。Excelで開いた際に日本語が文字化けしないよう、
CSV本体の先頭にUTF-8のBOM（`U+FEFF`）を付与している。

## 13. 「単語テスト名」（`state.getCurrentTestName()`）とタイトル欄の自動入力（Ver2.8）

12章のCSVファイル名と、テスト条件設定の「タイトル（任意）」欄の初期値は、どちらも
同じ「単語テスト名」という概念を初期値に使う。実装を重複させないよう、
`state.getCurrentTestName()`を両者共通の唯一の取得口にしている。

### 13.1 「単語テスト名」の実体（`state.js`の`getCurrentTestName()`）

Ver2.8時点では、`sourceFileName`（10章、教材名。Excelファイル名またはGoogleスプレッド
シートのタイトル）から、既知の拡張子（`.xlsx`/`.xlsm`/`.xls`/`.csv`）を除いた文字列を
返すだけの実装になっている。拡張子除去そのものは`utils/csv.js`の`stripKnownExtension()`
（12.2章）を呼ぶだけで、拡張子リストの実体は`utils/csv.js`側にしか持たない。

呼び出し側（`main.js`のCSV書き出し・タイトル自動入力）は、`state.getSourceFileName()`を
直接使わず、必ず`getCurrentTestName()`を経由する設計にしてある。これは、現時点では
「教材名（何を読み込んだか）」と「単語テスト名（今回のテストに付ける名前）」が同じ値だが、
将来的に両者が別概念になった場合（例えば、1つの教材から複数の異なる名前のテストを
作れるようにする等）でも、`getCurrentTestName()`の中身だけを差し替えればよく、
CSV書き出し・タイトル自動入力それぞれのコードを変更せずに済むようにするため。

### 13.2 タイトル欄の自動入力（`main.js`の`applyLoadedWords()` / `upgradeGoogleSheetTitle()`）

「タイトル（任意）」欄（`#testTitle`）は、教材の読込が成功するたびに
`formView.setTestTitle(state.getCurrentTestName())`で自動的にその単語テスト名を
書き込む。書き込みが発生するのは次の2箇所のみで、これ以外のイベント
（入力欄の変更・生成予定プレビューの再計算・レベル指定の変更等）ではタイトル欄に
一切触れない。

1. `applyLoadedWords()`（Excel/Googleスプレッドシートの読込成功時、および
   アプリ起動時の前回データ復元時に共通で通る箇所）。
2. `upgradeGoogleSheetTitle()`（Googleスプレッドシートの実際のタイトルが
   ベストエフォートで後から取得できた場合。10.3章と同じ非同期更新のタイミングで、
   「現在の教材」パネルの表示名を差し替えるのと同時にタイトル欄も更新する）。

この2箇所は、いずれも「新しい教材が読み込まれた」ことに起因するイベントであり、
それ以外のタイミングでタイトル欄の値を書き換えるコードが存在しないため、
「一度読み込んだ教材の範囲内でユーザーがタイトルを手動で書き換えた内容」は、
次に別の（あるいは同じ）教材を読み込み直すまでそのまま保持される
（要件「既にユーザーがタイトルを書き換えている場合は、その内容を勝手に上書きしない」
を、書き込みタイミングをこの2箇所だけに限定することで満たす設計。教材名が変わって
いなくても再読込するたびに新しい値で上書きする、シンプルな実装を採用している。
「同じ教材を再読込したときは書き換え前の内容を保つ」という、より細かい区別は
Ver2.8時点では行っていない）。

アプリ起動直後の前回データ復元フローでは、`applyLoadedWords()`によるタイトル自動入力の
直後に`restorePersistedSettings()`が前回保存した設定（`wordTestApp.lastSettings.v1`の
`testTitle`。ユーザーが手動で書き換えていた場合はその内容を含む）でタイトル欄を
上書きするため、最終的にはそちらが優先される（7章・9章と同じ、既存の設定復元の
順序に乗るだけで、この自動入力機能のために特別な分岐は必要なかった）。
